Boppo provides a WASM API to program new activities, interactive content, and games that run directly on Boppo.

To get started developing your own activities quickly its best to read the documentation for your language of choice:

* Rust: [boppofun/rust_boppo_wasm](https://github.com/boppofun/rust_boppo_wasm)
* AssemblyScript: coming soon
* If your preferred langauge does not exist and it has the ability to compile to WASM modules suitable for an ESP32
[let us know] your interested in a language binding or feel free to build your own using the WASM API below.

## WASM API for Language Bindings

The following APIs are meant to be used by language binding libraries. For activity developers please read the API
documentation for your language which will wrap the following in a more friendly API.

The Boppo tablet WASM runtime is WASIp1 compatible but not all WASI features are supported (e.g. filesystem APIs are
not yet available).

### Host Functions

#### boppo_wasm_poll(timeout_ms: i32) -> i64

If timeout_ms is less than 0, the function will block indefinitely until an
event occurs. If timeout_ms is greater than 0 the function will return
after the timeout if no host event occurred first. If timeout_ms is 0, the
function will check for events and return immediately if none are available.

Returns a HostEvent represented as an i64. The least significant byte is the
event type, and the remaining bytes are event-specific data.

Event types:

* 0: **Exit**

    The WASM activity should exit gracefully (i.e. return from the _start/main() function).

    Payload: None

* 1: **Button**

    A top button has been pressed or released.

    Payload: The two *most* significant bytes are a 2 byte payload. Of those 2
    bytes, the least significant 4 bits are the button index of the event. The
    next 10 bits are the current pressed state (1 is pressed) of all ten 10
    buttons (in index order). The 15th bit is if the current event is a press
    event (1) or a release event (0).

* 2: **Audio**

    An audio playback event occurred. The second byte is the audio sub-event type,
    and the following 4 bytes are the audio handle (i32, little-endian) the event refers to.

    Sub-event types:
    * 0: **Finished** — the audio clip finished playing. The activity should call
      `boppo_unload_audio` to free the associated resources.
    * 1: **BadHandleError** — a call to `boppo_play_audio` was made with a handle
      that does not exist or has already been unloaded.

* 3: **Timeout**

    The timeout_ms has elapsed without another event occurring.

    Payload: None

#### boppo_set_and_flush_lights(framebuffer_colors: *void)

Sets all 40 of the device lights and flushes the lights to the hardware.

The passed in pointer must point to a buffer of 120 bytes. Each light is represented by a 3 byte color value (RGB).

The ordering of the lights is the same as defined in the Rust [Lights struct](https://docs.rs/boppo_core/latest/boppo_core/struct.Lights.html#ordering)

#### boppo_open_audio_file(id: i32, path_ptr: *const u8, str_length: usize) -> i32

Open an audio file and associate it with a caller-provided integer handle `id`.
The path must be a UTF-8 string passed as a pointer (`path_ptr`) with its byte length (`str_length`).
The path is relative to the activity's directory (the folder containing the `.wasm` file).
Absolute paths are accepted and treated as relative by stripping the leading `/`, matching
the WASI file mapping convention.

Returns 0 on success, or a negative error code on failure. The `id` can then be used with the
other audio functions to control playback.

#### boppo_play_audio(handle: i32)

Starts playback of the audio loaded under `handle`. If the audio was previously paused,
calling this function resumes it. If the audio is already playing, this call is a no-op.

When playback finishes naturally, an **Audio/Finished** host event is sent to the activity.
The activity should respond by calling `boppo_unload_audio` to free resources.

#### boppo_stop_audio(handle: i32)

Stops playback of the audio loaded under `handle`. The audio clip remains loaded and can
be replayed with `boppo_play_audio`.

#### boppo_unload_audio(handle: i32)

Stops playback (if in progress) and frees all resources associated with `handle`.
After this call the handle is invalid.

#### boppo_set_audio_parameter(handle: i32, parameter: i32, value: f32)

Sets a playback parameter on the audio loaded under `handle`. Supported parameter values:

* 0: **Pause** — pause or unpause the clip. `value` of `0.0` unpauses; any other value pauses.
* 1: **Volume** — set the playback volume. `1.0` is the default level; values above `1.0`
  amplify the signal.
* 2: **Speed** — set the playback speed multiplier. `1.0` is normal speed.

### Module Exports

#### _start() (required)

As defined by WASIp1 this is the function the runtime will call to start the WASM activity. When this function returns the runtime will return to the active menu.
